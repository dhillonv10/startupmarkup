Startup Markup Language: II

  • Intended audience:
    • To teach developers the basics of building a startup
    • Extend to a language that anyone can use
    • No programming experience required
    • Inspired by the best features from many programming languages
    • Heavy focus on validation and creating a state of mind where the end-user validates everything

Startup Markup Language: III

  • Object-oriented language
    • To have structure, we need tools of organization that provide that structure
    • We use objects
    • Objects are at core of guiding this behaviour-driven language

What's an object?

A virtual representation of a thing

Let's use an example

      
  // Create a resume paper object
  // How would you describe it to a machine? 
  
  function createResumePaper extends paper {
 
    this.color = white 
    this.size = A4
    this.burnable = true 
    
}

Object-orientation

  • Object-oriented language?


    • Objects like the last example are the core of this language
    • Objects organized into the most basic functional units called function
    • Declaring a function creates an object

Object-orientation: II

  • Declaring a function? 

          
  // Create a paper object
  // How would you describe paper to a machine? 
  
  function createResumePaper extends paper {

}


    • Declaration is done by the key-word function
    • createResumePaper is the name of the function declared

Object-orientation: III

  • Declaring a function?

    • An object is created upon declaring the function
    • To access the object created by the function, we use the key-word this
    • this allows us to access properties of the object we created 


          
  // Create a paper object

    this.color = white 
    this.size = A4
    this.burnable = true

Object-orientation: IV

  • Extending a function? 

    
  function createResumePaper extends paper {
}
    • The extends key-word offers base-level characteristics to an object so that you don't have to redefine them.
    • A resumePaper object is first a piece of paper, and then a resume paper
    • It's pointless to reinvent the wheel and describe that a paper first
    • So let's skip all of that and use some simple pre-defined notions

Object-orientation: V

  • Pre-defined notions

    • We talked about using objects to provide structure for thinking
    • The pre-defined notion in the last example was paper, that created our resumePaper object
    • Pre-defined notions in startup markup language are really just well-defined objects that we give to the user
    • These objects have characteristics that I think will capture the lean methodology

Object-orientation: VI

  • Pre-defined notions

    • More formally, an end-user will access the characteristics of these pre-defined notions or objects and assign them properties of a real startup to generate an experience where all aspects of their business model are captured in this validation-heavy and execution-focused language.

Object Library

  • Philosophy

    • By using objects, the workings of a startup are treated like an app
    • You code the app which is modular with each module tested
    • And instead of being compiled, the app actually only exists to impose structures and limitations on your mind and thinking
    • This I hope has the result of getting the end-users into a mode of using best-practices to execute their business model.

Object Library: Cycle

Object Library: Cycle

  • Loop design, where each number evolves to the next one
  • This could highlight growing importance as the numbers move from one to the next
  • Used universally in this startup markup language
  • Slight modifications make up the rest of the object library

Object Library: cyclicEvolution

  • First modification

Object Library: cyclicEvolution

  • Loop design, with an exit
  • Designed to show a resolution from iterative search with one event evolving to the next
  • Can be used to show recent trends leading up to a product idea
  • Every point or event can also be used as a meta-event to capture more cateogorical or relational events.

Object Library: cyclicEvolution

  • Meta-event?

Object Library: cyclicEvolution

  • A topic-subtopic approach in describing the events associated to the cyclicEvolution object
  • This is universal to all the cyclic objects in this language

Object Library: writePad

  • A simple modification of cyclicEvolution to list mutually independent topics but no evolution occurs between them
  • Ideas can be listed in any order here, the only structure imposed is that they are a part of the cycle
  • Think of this as branching
  • Still follow meta-events so using a cyclicEvolution object offers inherent structure in the process

Object Library: writePad

  • No structure

Object Library: cList

  • The only structure here that doesn't follow a cycle structure
  • Simple implementation of cornell notes as an object
  • One thing -> to another
  • Can be used to define your competitors and what they do
  • Competitor -> Product

Object Library: Event Loop

  • The single most important structure of this markup language
  • An event loop checks for events that are defined for the event loop
  • If any of the defined events in the event loop are triggered, a subroutine is initiated
  • A subroutine is basically a fancy way of saying one cause (event triggered) leads to an effect
  • The effect of an event loop is ALWAYS to run a unit test

Object Library: Event Loop

  • The simplest unit of work in startup markup is a function
  • Every function has a unit test associated with it, to test the function
  • Every single unit test is basically a customer development or validation check
  • Event loop continously runs and unlike cyclicEvolution, this doesn't have an exit

Object Library: Event Loop

  • Instead of organizing topics and sub-topics, event loop monitors certain files for change
  • These files are the critical aspects of your business model written in startup markup
  • If any of the files are changed, event loop stops and will jamm the whole application
  • While the application is jammed, a subroutine is initiated to test the the change that was introduced in one of the core files
  • This enforces the theme of small, managable and easy to check changes done one at a time to the business model
  • This has very fundamental implications for how startups execute their business model

Object Library: Event Loop

  • Realize that this is not a compiled language, there really is no event loop that will stop you from doing anything
  • This exercise is nothing more than a mental constrain created to help understand business model execution done with programmatic logic
  • After the subroutine ran and the change was tested, if it returns a positive value, the event loop starts again and checks for if there are any more changes in any other files.
